# web/default 前端优化报告

> 目标范围：仅 `web/default`。基线数据来自本机生产构建（`bun run build`，rsbuild/rspack，2026-07-05）。
> 结论一句话：**栈很现代、路由级分包已开，但首屏 ~1.36 MB gzip、单个入口块 846 KB gzip，且攒了多个死依赖 + 数个 3–6 MB 的巨型路由块，从未接过打包分析器。**

---

## 一、摘要（Top 5）

1. **首屏过重**：初始加载 ≈ **1.36 MB gzip**（JS 1.30 MB + CSS 63 KB），其中入口 `index.js` 单块就 **846.6 KB gzip / 3.04 MB raw**。合理目标是初始 JS < 300–500 KB gzip，现状约 2.6–4×。
2. **数学/markdown 被打进首屏**：入口块与初始共享块里含 `katex`、`marked`、`dompurify`（富文本渲染栈），首屏用不到却被下载。
3. **巨型路由块**：109 个异步块中 13 个 ≥ 300 KB gzip，最大 `2280` 块 **2.75 MB gzip / 6.83 MB raw**。已确认 `6956`=VChart、`8686`=AI SDK+厂商图标；其余 5 个 3–6 MB 的块连指纹都识别不出——**正说明必须接分析器**。
4. **死依赖**：`react-globe.gl`、`@tanstack/react-virtual` 全树 0 引用；`recharts` 只被无人引用的 `components/ui/chart.tsx` 拉着。另有 `@xyflow/react`、`embla-carousel-react` 错放在 devDependencies。
5. **后台大表零虚拟化**：67 个文件用 `@tanstack/react-table`，**0 个**用 `react-virtual`（依赖已装、AGENTS.md 3.4 已要求）。上千行的 usage-logs / users / channels 全量渲染 DOM。

### 2.2 最重的异步（路由/懒加载）块 —— 已归因

> 归因方式：一次性 `NO_MINIFY=true` 生产构建，读未压缩产物里的 `node_modules/…` 模块路径（压缩产物的字符串指纹不可靠，5 个最大块曾无法识别）。**结论：体积主体全是「按需图标集」，不是首屏、不是图表引擎。**

| `async/9907` | 5371.6 KB | 1203.9 KB | **react-icons 整套**（Phosphor `pi` 等，含 `PiPresentationChart…`） |
| `async/8686` | 5139.5 KB | 975.3 KB | **@lobehub/icons ＋ 传递依赖 antd/antd-style**（厂商 logo） |

| `async/9858` | 3164.3 KB | 374.1 KB | **react-icons 整套**（gz 比高＝同构图标数据） |
| `async/3290` | 2119.8 KB | 560.8 KB | **react-icons 整套** |
| `async/6956` | 2060.7 KB | 560.1 KB | **@visactor/VChart**（唯一 vchart 块，已去重共享） |
| `async/2633` | 2147.8 KB | 478.0 KB | **react-icons 整套** |
| `async/5893` | 1961.9 KB | 418.0 KB | **react-icons 整套** |

- **总产物：58.5 MB raw / 16.9 MB gzip**。其中 **~9 个 react-icons 整套图标集，合计约 30 MB raw**，是总体积的绝对主体。
- **关键真相**：这些巨块**全部是异步/按需**——`react-icons` 集只在付款/钱包页用到某个图标时才拉整套；`@lobehub/icons` 只在渠道/模型/定价等页出现厂商 logo 时加载。**它们不进首屏**。首屏（entry `index.js` 842gz）实测是 `lucide-react + zod + @base-ui + 应用代码`，属正常量级、难安全再削。
- 因此「减总体积」的最大杠杆是 **react-icons 按名加载器** 与 **@lobehub/icons(antd)**，两者都需产品级取舍（见第六节 P1）。手工指纹不可靠、`rsbuild performance.bundleAnalyze` 在本版本未产出报告，故归因靠 `NO_MINIFY` 构建拿到，建议长期接 `@rsbuild/plugin-rsdoctor`。

## 三、优化项（按优先级）

### P0-A｜首屏瘦身（收益最大、面向公网访客）

- **把富文本渲染栈移出首屏**：`katex`（含字体）、`marked`、`dompurify` 目前落在入口/初始共享块。让 `@/components/rich-content` 及其 markdown/math/sanitize 依赖走 `lazy()` + 动态 `import()`，仅在真正渲染富文本时加载。预计可从首屏削掉约 **150–250 KB gzip**（katex 是大头）。
- **拆解 846 KB 的入口块**：入口块过大通常是「本该按需的东西被同步 import 进了 App 根/布局」。需分析器逐模块看，重点排查 app-shell、布局、`__root` 上的同步导入。
- **落地页 `/` 首屏**：`features/home/index.tsx` 同步 `import { PremiumLanding }` → 后者同步 `import { PremiumWorldMap }`（`dotted-map`）并拉入 `gsap` + `lenis`。目前只有 `EnergyCore`（three.js）是 `lazy` 的（`premium-hero.tsx:28`）。把 worldmap 和首屏以下区块按同样方式懒挂，`gsap/lenis` 随之延后。

### P0-B｜死依赖 / 依赖错配清理（近零风险）

| 项 | 证据 | 动作 |
| --- | --- | --- |
| `react-globe.gl` | 全树 0 引用 | 从 `dependencies` 移除 |
| `@tanstack/react-virtual` | 全树 0 引用 | 见 P1-B（本该启用，先留） |
| `recharts` | 仅 `components/ui/chart.tsx` 引用，而该文件 0 引用 | 删文件 + 移除依赖（图表已统一 VChart） |
| `@xyflow/react` | 7 个源文件在用，却在 devDependencies | 移到 `dependencies` |
| `embla-carousel-react` | 1 个源文件在用，却在 devDependencies | 移到 `dependencies` |

> 注：未被 import 的依赖本就不进产物，删除主要收益是安装体积/审计面/`knip` 干净，并防止将来被误引入。`package.json` 已有 `bun run knip` 脚本，建议先跑一次拉全死代码/死导出清单。

### P1-A｜巨型异步块归因与拆分（**先接分析器**）

1. **接入分析器**（本报告第一执行项）：加 `@rsbuild/plugin-rsdoctor` 或用 rspack `stats.json`，得到 chunk→module 精确体积。手工指纹已证明不可行。
2. 用分析器确认 5 个未识别的 3–6 MB 块归属（强候选：VChart 引擎 vrender/vgrammar、shiki 语法、three/globe、codemirror、xyflow、katex 字体、`@lobehub` 全量图标、`tokenlens` 模型数据）。
3. 按归属做**块内二次拆分 / 按需引入**，典型手法：
   - **shiki**：改用 `shiki/core` + `createHighlighterCore` + 仅注册实际用到的语言/主题，禁用 `shiki/bundle/web`（全语言）。
   - **VChart**：按需引入图表类型而非整包；图表组件已 `lazy`（dashboard），确保 rankings/pricing 的图也走懒加载。
   - **three / dotted-map**：限定在落地页懒块，勿泄漏到共享块。

### P1-B｜后台大表虚拟化（运行时渲染收益）

- 67 处 `@tanstack/react-table`、0 处 `react-virtual`。给可能上千行的表（usage-logs、users、channels、log、redemption-codes）接 `@tanstack/react-virtual`（依赖已装）。这是滚动/渲染流畅度收益，非包体收益，但用户可感。

### P2｜收敛与缓存（低优先）

- **图标库 4 套**：`lucide-react`(280) 事实标准、`@lobehub`(12) 厂商 logo 合理、`@hugeicons`(21) 规范内、**`react-icons`(14) 离群** → 收敛到前者后移除，减一套体系。
- **splitChunks 缓存**：当前只分了 `react / ui-primitives / tanstack`。可加 `vendor-charts`（VChart）cacheGroup，提升 dashboard 子路由间长缓存（VChart 已 async 分包，这是缓存优化非体积）。
- **字体**：`@fontsource-variable/lora` + `public-sans` 全量打入，确认按需子集/字重、`font-display: swap`。
- **动画双栈**：`motion`（应用壳，7）与 `gsap+lenis`（仅落地页，5）按页面隔离、重叠小，除非要统一否则维持现状。

## 四、执行顺序（每步后 `typecheck` + `lint` + 生产构建对比体积）

1. **接分析器 + 跑 `knip`**，拿到 chunk→module 基线与死代码清单（不改业务代码）。
2. **P0-B 死依赖/错配清理**（`react-globe.gl`、`recharts`+`chart.tsx`、xyflow/embla 归类），构建确认无回归。
3. **P0-A 首屏瘦身**：富文本栈懒加载 + 落地页 worldmap 懒挂，重测首屏 gzip。
4. **P1-A 巨型块拆分**：按分析器归因，优先 `2280`(2.75 MB)、`3444`(1.97 MB)、`9907`(1.20 MB)。
5. **P1-B 大表虚拟化**：先做 usage-logs / users 两个最大表验证收益，再铺开。
6. **P2 收敛项**：`react-icons` 收敛、`vendor-charts` 缓存组、字体子集。

## 五、方法与局限

- **数据来源**：`rm -rf dist && bun run build` 的 rspack 产物体积（raw + gzip 均来自构建器自报）。
- **归因方式**：对压缩后的 chunk 做库特征字符串指纹 + 源码 import 定位。**局限**：5 个最大异步块经充分特征扫描仍无法归因（压缩/属性混淆所致），故精确 chunk→module 归属必须依赖分析器；本报告对未确认项均标注「未识别 — 待分析器」，不臆断。
- **首屏合计**为所有非 `async/` 前缀 JS + 入口 CSS 之和，即 HTML 入口加载的初始块组。

## 六、执行结果（2026-07-05，已落地并验证）

> 全程每步 `bun run build` 实测；收尾 `bun run typecheck` **0 error**，改动文件 lint/format/copyright 全绿。仓库既有的 ip-ban / section-registry / main.tsx 等 lint/type 报错为**存量债**，与本次无关，未触碰。

### 6.1 已完成

**P0-B 死依赖清理（knip + 精确 import 追踪双重确认，全部零引用）**
- 移除 8 个依赖：`react-globe.gl`、`next-themes`、`recharts`、`react-resizable-panels`、`auto-skeleton-react`、`tokenlens`、`@xyflow/react`、`embla-carousel-react`。
- 删除 13 个孤儿脚手架文件：`ui/{chart,resizable,carousel}.tsx`、`auto-skeleton.tsx`、`ai-elements/{context,inline-citation,canvas,node,edge,panel,controls,connection,toolbar}.tsx`（整套 xyflow 流程画布 + 未接入的 shadcn 组件，均无任何引用者）。
- `@tanstack/react-virtual` 虽也零引用，但**保留**——它本应被用于大表虚拟化（见 6.2）。

**P0-A 首屏瘦身（关键路径，非总量）**
- **🔑 i18n 语言包按需加载（最大单项）**：`i18n/config.ts` 原本 `import` 6 个语言 JSON 全塞进 `resources`，实测入口 `index.js` 里塞了 6 套翻译（~2.5 MB raw，含 141,932 个西里尔字符 = 整套俄语），是入口块 ~85% 的体积。改为**每语言 `import()` 独立分包**，只加载「当前语言 + en 兜底」，`main.tsx` 渲染前 `await i18nReady` 保证首绘即完整翻译。入口 `index.js` 从 **842.9 → 162.7 KB gz（−680）**。
  - 修复：react-i18next 默认只在 `languageChanged` 重渲染、不在 `addResourceBundle`（'added'）重渲染，故切到未加载语言会停在兜底文案 → 加 `react.bindI18nStore: 'added'` 解决。
- **富文本渲染栈懒加载**：`components/ui/markdown.tsx` 拆为 `markdown-impl.tsx`（marked+katex+katex CSS+dompurify，789 行）+ `React.lazy` 薄壳。根因是 `NotificationPopover` 在两个 header（全站外壳）里渲染 `RichContent → Markdown`，把整套数学/markdown 顶进首屏。三处消费者零改动（Suspense 内置在壳里）。
- **落地页 worldmap 懒挂**：`premium-landing.tsx` 把 `PremiumWorldMap`（`dotted-map`）改 `React.lazy`，`/` 路由首绘不再等这块首屏下方的地图。
- **@lobehub/icons 懒加载**：`lib/lobe-icon.tsx`（`getLobeIcon` 用 `import * as` 全量命名空间 + 动态取名，无法 tree-shake）拆出 `lobe-icon-impl.tsx` 走 `React.lazy`，`getLobeIcon(name,size)` 签名不变、13 处调用零改动；厂商 logo 块（含 antd）从「路由阻塞」变「按需」。

### 6.2 待决策 / 未落地（含理由）

**P1 · 减总量的最大两个杠杆（需产品取舍，故未擅自改）**
- **react-icons 按名加载器**（`components/react-icon-by-name.tsx`）：为「按字符串名渲染任意图标」注册了 ~31 个图标集的动态 import，每用到一个图标就拉**整套**（单套 2–7 MB），是总体积主体（~30 MB raw）。仅付款/钱包 3 处用到。建议：改用 `@react-icons/all-files`（单图标级 import）或收敛为「受支持图标白名单 + 兜底」。**会改变管理员可任意配置图标名的行为，须产品确认。**
- **@lobehub/icons 拖入 antd**：8686 块 975 KB gz 里 antd/antd-style 占大头。已做懒加载缓解；进一步需评估能否只引 logo 子集、去 antd 依赖。

**P1-B · 大表虚拟化（高价值但需可视化验收，未盲改）**
- 67 处 `@tanstack/react-table`、0 处虚拟化；共享 `components/data-table/core/data-table-view.tsx:281` 是 `rows.map(...)` 全量渲染。**在此一处接 `@tanstack/react-virtual` 即可惠及全部 67 张表**。未落地原因：该表含列固定(pinning)/列宽/移动端卡片布局，虚拟化与 sticky 列交互复杂，**须跑起前端+后端+数据做可视化验收**才能确认不破版；本环境无运行中后端与数据，盲改风险高于收益。建议按行数阈值（如 >100 行才虚拟化）灰度接入并逐表目视验收。

**P2 · 收敛（低优先）**
- `react-icons` 的 3 处静态 `si` 图标 import（`SiGithub` 等）可 tree-shake，无需处理；问题只在上面的动态加载器。
- `vendor-charts` cacheGroup：VChart 实测已是**单个共享块**（无重复），加 cacheGroup 无体积收益，略。
- 字体（lora/public-sans）子集化、`react-icons` 与 `@hugeicons` 图标体系合并：仍可做，收益有限。

### 6.3 顺带产出
- 顺手证伪：`shiki` 全站仅 `import type`（代码高亮实为 CodeMirror），**不进运行时产物**，原报告「shiki→core」一项作废。
- 归因方法沉淀：`NO_MINIFY=true bun run build` 读未压缩产物的 `node_modules/…` 路径，是本仓库最可靠的 chunk→库归因手段（`performance.bundleAnalyze` 在当前 rsbuild 版本未产出报告）。
